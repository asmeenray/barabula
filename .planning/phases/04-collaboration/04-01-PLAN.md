**6. Postgres trigger — auto-convert pending invites when a new user registers:**
```sql
CREATE OR REPLACE FUNCTION public.handle_new_user_invites()
RETURNS trigger
LANGUAGE plpgsql
SECURITY DEFINER
SET search_path = ''
AS $$
BEGIN
  -- Convert any pending invites matching the new user's email to collaborator rows
  INSERT INTO public.collaborators (itinerary_id, user_id, role)
  SELECT itinerary_id, NEW.id, role
  FROM public.invites
  WHERE invitee_email = NEW.email
    AND accepted_at IS NULL;

  -- Mark the converted invites as accepted
  UPDATE public.invites
  SET accepted_at = NOW(), accepted_by = NEW.id
  WHERE invitee_email = NEW.email
    AND accepted_at IS NULL;

  RETURN NEW;
END;
$$;

CREATE TRIGGER on_user_created_match_invites
  AFTER INSERT ON public.users
  FOR EACH ROW
  EXECUTE PROCEDURE public.handle_new_user_invites();
```

Critical notes:
- Use `(SELECT auth.uid())` (correlated subquery form) everywhere — not bare `auth.uid()`. This prevents RLS infinite recursion per Supabase docs.
- The trigger fires on `public.users` AFTER INSERT, which is AFTER the existing `on_auth_user_created` trigger has already inserted the user row. So `NEW.email` is available.
- Do not add the Phase 4 SQL to the existing migration blocks — keep it as a clearly labelled append so the file remains the source-of-truth for both Phase 2 baseline and Phase 4 additions.
