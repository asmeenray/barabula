    **Pitfall 1: Chat layout not overriding parent max-w-6xl**

    Read `src/app/(authenticated)/layout.tsx`. The `<main>` tag has `max-w-6xl mx-auto px-4 py-8`.

    The `fixed inset-0` in chat/layout.tsx should work because fixed elements escape the document flow. However, the parent layout's `<nav>` will still render and the fixed chat div will cover it. Test this:
    - If the nav is visible at the top of the chat page (desired: hidden on chat — full immersion per CONTEXT.md "50/50 split" with no nav chrome), verify the z-index on chat/layout.tsx covers the nav.
    - If the nav leaks through: bump z-index from z-10 to z-30.

    Correct chat/layout.tsx should be:
    ```tsx
    export default function ChatLayout({ children }: { children: React.ReactNode }) {
      return (
        <div className="fixed inset-0 z-30 bg-white">
          {children}
        </div>
      )
    }
    ```
    The `z-30` clears both the nav (z-0) and any other stacking contexts. `bg-white` prevents any bleed-through.

    **Pitfall 2: Video element shows black frame before poster loads**
    In VideoHero.tsx, the `<section>` already has `bg-slate-900` as fallback. If `poster="/images/hero-poster.jpg"` causes a 404 console error:
    - Either create a minimal placeholder at `public/images/hero-poster.jpg`
    - Or remove the `poster` attribute from the video element and rely on `bg-slate-900` fallback

    Check if `public/images/hero-poster.jpg` exists. If not: remove the `poster` attribute from the video tag in VideoHero.tsx. Add a comment: `{/* poster omitted — bg-slate-900 fallback covers initial load */}`.

    **Pitfall 3: DestinationCards using Unsplash URLs may be slow or fail**
    The DestinationCards component uses Unsplash image URLs. These load from an external CDN. This is acceptable for development but may be slow.

    Check if Next.js image optimization complains about external URLs (it won't since we're using `<img>` not `<Image>`). However, the `eslint-disable` comment is needed per the DestinationCards code.

    If the images fail to load (network-dependent), the card still renders correctly with broken image placeholder — acceptable for development.

    **Pitfall 4: `text-sand` color in HowItWorks may not resolve if tailwind config scan misses it**
    In HowItWorks.tsx, `text-sand` references the `sand` color token added to tailwind.config.ts in Plan 01.

    Run `npm run build` — if `text-sand` produces a warning or the build shows unknown class: verify tailwind.config.ts has `colors: { sand: '#C97B4B' }` in theme.extend. If missing, add it.

    **Pitfall 5: DM Serif Display may not apply correctly**
    The `font-serif` class should resolve to DM Serif Display via the CSS variable chain set in Plan 01.

    Verify: `tailwind.config.ts` has `fontFamily: { serif: ['var(--font-dm-serif)', 'Georgia', 'serif'] }`.
    Verify: `layout.tsx` has `className={`${inter.variable} ${dmSerifDisplay.variable}`}` on `<html>`.

    If the variable chain is broken (font shows as Georgia fallback instead of DM Serif), the chain is broken. Fix by ensuring both variables are on `<html>` not `<body>`.

    **Pitfall 6: QuickActionChips shows during history loading**
    In chat/page.tsx, `QuickActionChips disabled={sending || historyLoading}` should prevent display while loading. Verify the disabled prop propagates correctly — when `disabled` is true, the component returns `null` (per the implementation). If chips flash briefly on load before history loads: the disabled guard handles it.

    **Apply all identified fixes.** Run build after each fix to confirm it's clean.

    Leave the authenticated layout unchanged.

The `opacity-40` on the image + `bg-gray-950/60` overlay should produce a subtle textured dark background. If it's too dark (pure black): reduce overlay to `bg-gray-950/40`. If too bright (image dominates): increase overlay to `bg-gray-950/70`.

    Target: the ambient panel should feel like a tasteful dark background with a hint of a destination scene visible, not a photo collage.

    3. In `src/components/landing/VideoHero.tsx`, verify the scroll hint at the bottom reads `Scroll to explore` in white/40 text. If it's missing or misaligned: confirm `absolute bottom-8 left-1/2 -translate-x-1/2` is correct.

    4. Ensure `src/app/globals.css` has `html, body { height: 100%; }` or equivalent so `h-screen` works correctly on all browsers. Without this, `h-screen` may not fill the viewport on some mobile browsers.

    Add to globals.css if not present:
    ```css

      height: 100%;
    }
    ```
